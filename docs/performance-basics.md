# パフォーマンス基礎: Big-O 記法と実行時間

このドキュメントはリポジトリ全パターンの前提知識として、「なぜループが遅くなるか」「計算量とは何か」を 1 ページで解説します。

**想定読者**: Big-O を学校で習った記憶はあるが、コードレビュー時に瞬時に判定できない方。  
**スコープ**: 漸近計算量の直感のみ。数学的証明や厳密な記法は扱いません。

---

## なぜ実行時間は入力サイズで増えるか

配列 1 件を処理するコストはほぼ一定です。しかし n 件を処理するなら、そのコストは「1 件あたりのコスト × 件数」になります。

問題が起きるのは **「1 件あたりのコスト」自体が件数に依存する** ときです。

```
n = 100  のとき: 100 × 100 =      10,000 回の操作
n = 1,000 のとき: 1,000 × 1,000 = 1,000,000 回の操作（100 倍の増加で 100 倍ではなく 10,000 倍）
```

n が 10 倍になっても実行時間が 10 倍で済むか、100 倍になるかは、「1 件あたりのコストが n に依存するかどうか」で決まります。

---

## Big-O 記法とは何か

Big-O は「n が十分大きいとき、実行時間の増え方を支配する項だけを書く」記法です。

- `5n² + 100n + 3000` → `O(n²)` — n が大きくなると n² の項が圧倒的になる
- `n + log n` → `O(n)` — n の項が支配的

**定数倍を無視する理由**: 実行時間の定数倍はハードウェアや実装の差で容易に変わります。アルゴリズムの本質的な「増え方の形」を比較するために無視します。

Big-O は「最悪ケースの上界」を表すことが多く、「このコードは最悪でも n に比例したステップ数で終わる」と読みます。

---

## 代表的な計算量クラス

| 記法 | 意味 | 代表的な操作 | n を 10,000 倍にすると実行時間は |
|---|---|---|---|
| O(1) | 入力サイズに依存しない | `map.get(key)` / `obj[key]` | 変わらない |
| O(log n) | 入力が 2 倍になっても +1 ステップ | 二分探索 / ヒープ操作 | 約 13 倍（log₂ 10,000 ≒ 13） |
| O(n) | 1 回走査 | `for...of` / `arr.map()` | 10,000 倍 |
| O(n log n) | ソートの典型 | `arr.sort()` / マージソート | 約 130,000 倍 |
| O(n²) | 二重ループ | ループ内 `find` / `filter` | 100,000,000 倍 |

O(n²) と O(n) の差は n が大きいほど壊滅的になります。n=10,000 の場合、O(n²) は O(n) の 1 億倍になりえます。

---

## ループのネストと「掛け算」のメンタルモデル

ループが重なると計算量は **掛け算** になります。

```
外側 n 回 × 内側の各回の計算量 = 全体の計算量
```

ループ内で配列を `find` / `filter` / `some` / `includes` で走査すると、内側が O(n) になります:

```
外側 n 回 × 内側 n 回 = O(n²)
```

レビュー時の視点: **ループの内側に、別の配列の全走査が隠れていないか？**

ループ変数に依存しない処理（ loop-invariant ）はループの外に出せます。例えば `JSON.parse(jsonString)` が固定文字列を対象にしているなら、ループ外で 1 回だけ実行すれば O(n × parse) → O(parse + n) になります。

このメンタルモデルが実際に現れるパターン:

- [ループ内 線形探索](../patterns/loop-linear-search/) — ループ内の `find`/`filter` が O(n)
- [ループ内 includes → Set](../patterns/loop-includes-to-set/) — ループ内の `includes` が O(n)
- [reduce + スプレッド](../patterns/reduce-spread/) — `reduce` 内のスプレッドが O(n)
- [ループ内 JSON.parse](../patterns/json-parse-in-loop/) — ループ不変処理の繰り返し

## 計測 vs 漸近計算量

漸近計算量は「n が十分大きいとき」の話です。n が小さければ定数倍が逆転することがあります。

```
O(n²) のコードが n=10 なら 100 ステップ
O(n) でも Map 構築コストで n=10 では遅いことがある
```

**計算量がよくても必ずベンチマークで確認してください**。このリポジトリの各パターンには実際に手元で動かせるベンチマークコードが付いています: [CONTRIBUTING.md#計測ヘルパー](../CONTRIBUTING.md#計測ヘルパー)
